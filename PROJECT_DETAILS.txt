1. PROJECT SUMMARY
==================
This repository is a Python-based computer vision prototype for smart traffic monitoring and dynamic traffic signal timing.

The main idea:
- A YOLO object detection model analyzes video/camera frames.
- The edge inference script counts detected vehicle classes.
- The backend receives detections and stores them in MongoDB.
- The backend computes traffic signal timing based on weighted vehicle counts.
- A Streamlit dashboard displays live counts, signal health, process status, computed timings, and alerts.
- Edge heartbeat and watchdog scripts monitor runtime health and report failures.

The project currently mixes production-intended prototype code, older experiments, datasets, trained model weights, and unrelated small scripts.

Main product names used in code:
- SmartFlow
- AutoRoute
- Traffic Signal Timing System


2. HIGH-LEVEL ARCHITECTURE
==========================
Core runtime flow:

1. MongoDB starts locally through Docker or Docker Compose.
2. FastAPI backend starts on http://127.0.0.1:8000.
3. Edge inference script loads a YOLO model and reads video/camera frames.
4. YOLO produces detections with class id, confidence, and bounding box coordinates.
5. The inference script converts detections into per-class counts.
6. The inference script posts detections to POST /detections.
7. Backend stores detections in MongoDB collection autoroute.detections.
8. Dashboard polls backend endpoints every 5 seconds.
9. Dashboard shows latest counts, signal health, computed timings, process status, and alerts.
10. Heartbeat script posts hardware/runtime metrics to POST /heartbeat.
11. Watchdog checks backend, inference, and heartbeat processes and restarts/report alerts when they fail.

Main components:
- backend/: FastAPI API server, MongoDB persistence, timing algorithm, alert handling.
- dashboard/: Streamlit dashboard.
- ai/: YOLO inference experiments, model weights, training/output artifacts, datasets.
- edge/: runtime edge scripts for inference, heartbeat, watchdog.
- infra/: MongoDB Docker Compose file.

5. BACKEND API DETAILS
======================
Base URL:
- http://127.0.0.1:8000

GET /
- Health/root endpoint.
- Returns message that SmartFlow backend is running.

POST /detections
- Receives AI detections from edge device.
- Expected JSON:
  {
    "junction_id": "J1",
    "ts": 1710000000.0,
    "detections": [
      {"cls": 2, "conf": 0.91, "xyxy": [10, 20, 100, 200]}
    ],
    "counts": {"car": 3, "bus": 1}
  }
- Stores document in autoroute.detections.
- Timestamp is converted from Unix seconds to UTC datetime.

GET /latest/{junction_id}
- Returns latest detection counts for a junction.
- Used by dashboard for live counts.

POST /heartbeat
- Receives edge health metrics.
- Expected JSON:
  {
    "junction_id": "J1",
    "ts": 1710000000.0,
    "cpu": 40.5,
    "mem": 62.0,
    "fps": 12.3,
    "avg_conf": 0.77,
    "camera_ok": true
  }
- Stores document in autoroute.heartbeats.

GET /status/{junction_id}
- Reads latest heartbeat and returns status:
  - OK if last heartbeat is <= 15 seconds old.
  - DEGRADED if last heartbeat is <= 45 seconds old.
  - OFFLINE if last heartbeat is older than 45 seconds.
- Sends SMS when offline. This can spam SMS if the endpoint is polled repeatedly.

POST /compute_timing
- Computes signal timing from counts per approach.
- Expected JSON:
  {
    "junction_id": "J1",
    "approaches": {
      "N": {"car": 4, "bus": 1},
      "S": {"bike": 6},
      "E": {},
      "W": {"truck": 1}
    }
  }
- Stores result in autoroute.timings.
- Returns cycle_length and per-approach phases.

POST /process_status
- Accepts process state posted by watchdog-like systems.
- Stores/upserts into autoroute.processes.
- Current dashboard does not use stored process status; it calls live psutil status endpoint instead.

GET /process_status/{junction_id}
- Checks local machine process list for inference3.py and heartbeat.py.
- Returns running/stopped status.

POST /alert
- Stores alert in autoroute.alerts.
- Sends SMS using Twilio.

GET /alerts/{junction_id}
- Returns latest 20 alerts for the junction.
- Current alert writer may store "junction" while reader filters by "junction_id", so some alerts may not appear in dashboard.


6. SIGNAL TIMING LOGIC
======================
The current timing algorithm is in backend/main.py.

Vehicle weights:
- bike: 0.5
- car: 1.0
- bus: 2.5
- truck: 3.0
- pedestrian: 1.0
- unknown classes default to 1.0

Constants:
- MIN_GREEN = 5.0 seconds
- MAX_GREEN = 60.0 seconds
- BASE_CYCLE = 30.0 seconds
- K = 30.0
- MAX_CAPACITY_PER_APPROACH = 30.0
- YELLOW_TIME = 3.0 seconds
- ALL_RED = 1.0 seconds

Algorithm:
1. For each approach, convert object counts into weighted traffic load.
2. Sum total weighted load.
3. If total load is zero, split BASE_CYCLE equally across approaches.
4. Otherwise compute cycle length:
   BASE_CYCLE + K * (total_load / (number_of_approaches * MAX_CAPACITY_PER_APPROACH))
5. Effective green time excludes yellow and all-red for every approach.
6. Each approach gets green time proportional to its weighted load.
7. Green time is clamped between MIN_GREEN and MAX_GREEN.

Limitation:
- If every approach receives the same global count object, the dashboard computes identical timings for N/S/E/W. True lane or approach assignment is not implemented yet.

10. CURRENT ISSUES AND RISKS
============================
Security:
- Twilio SID/auth token and phone numbers are hardcoded in backend/sms_utils.py.
- MongoDB username/password are hardcoded in backend/main.py and Docker files.
- Secrets should be rotated if they are real and moved to environment variables.

Dependency management:
- No requirements.txt or pyproject.toml was found.
- Reproducing the environment will be difficult on another machine.

Path consistency:
- edge/inference3.py uses MODEL_PATH = ai/runs/detect/train15/weights/best.pt, but this path was not visible in the top-level inventory.
- edge/inference3.py uses VIDEO_PATH = data/sample2.mp4, while the actual sample appears under ai/data/sample2.mp4.
- ai/inference3.py uses best.pt and sample2.mp4 relative paths that may fail unless run from a specific directory.

Schema consistency:
- Some older scripts post "junction" instead of "junction_id".
- Some older scripts omit ts and counts, so they do not match current backend DetectionPayload.
- /alert stores "junction" from incoming payload, but /alerts/{junction_id} filters by "junction_id"; this can hide alerts from the dashboard.

Dashboard logic:
- compute_timing sends the same count dictionary to N, S, E, and W, so the dashboard cannot compute true per-direction timing yet.
- Dashboard backend URL and junction id are hardcoded.

SMS behavior:
- /status/{junction_id} sends SMS every time an OFFLINE status is detected.
- Because dashboard auto-refreshes every 5 seconds, this can repeatedly send SMS.

Process status:
- GET /process_status/{junction_id} checks local process list on the backend machine, not necessarily the actual edge device.
- POST /process_status stores process data, but the dashboard currently does not read it.

Data/model organization:
- The repo contains multiple datasets, model weights, archives, and experiments together.
- Large files can make Git operations and backups slow.
- Some datasets/classes are not aligned with traffic detection.

Code quality:
- Several files are older experiments and are not compatible with the current backend.
- ai/utils.py has an unfinished map_to_lane function.
- ai/test2.py has an argparse bug.
- Readme.md is not a real setup guide.
- There are no automated tests for API behavior, timing logic, or payload compatibility.

Production readiness:
- No authentication on backend endpoints.
- No CORS policy shown for current backend.
- No structured logging.
- No retry/backoff strategy for backend posts.
- No message queue for buffering when backend is down.
- No camera reconnection strategy beyond exiting the inference loop.
- No formal deployment packaging for backend/dashboard/edge services.

15. QUICK PROJECT DESCRIPTION FOR PRESENTATION
==============================================
SmartFlow is a computer vision based smart traffic signal prototype. It uses YOLO object detection to identify and count vehicles from traffic camera footage, sends live detection results to a FastAPI backend, stores events in MongoDB, computes adaptive green-light timings using weighted vehicle counts, and visualizes real-time traffic, system health, alerts, and signal timings through a Streamlit dashboard. Edge scripts provide heartbeat monitoring and watchdog-based recovery so the system can detect camera or process failures and raise alerts.


16. FINAL NOTES
===============
The strongest part of this project is that it already connects AI inference, backend storage, dashboard monitoring, signal timing, and edge health checks into one working prototype concept.

The biggest gaps are reproducibility, secret handling, schema consistency, path consistency, and true per-lane traffic mapping. Fixing those will turn this from a collection of strong prototype pieces into a much cleaner end-to-end system.
